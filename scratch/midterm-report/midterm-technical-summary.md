# 毕设中期汇报技术总结

## 1. 汇报定位

本课题当前的中期工作，不应仅表述为“已经完成若干仿真脚本修改”，而应概括为：

1. 完成了面向 LEO-NTN 切换研究的三维物理模型设计。
2. 在此基础上，将场景逐步整理为可重复运行、可扩展、可统计的仿真平台。
3. 在平台上定义了当前阶段的传统 A3 baseline，并为后续改进策略预留了统一入口。

因此，中期汇报建议采用以下主线：

- 先讲技术上的设计。
- 再讲平台化、工程化设计。
- 然后讲当前选定的仿真 baseline。
- 最后讲下一步优化方向。

这条主线比“按时间顺序罗列修改记录”更适合中期汇报，因为它能清楚体现本课题已经从前期建模阶段推进到了可开展对比实验的平台阶段。

## 2. 课题目标与当前阶段定位

### 2.1 课题目标

本课题关注的是低轨卫星高速运动场景下的 5G-NTN 切换问题。核心目标不是单纯验证链路是否能建立，而是研究：

- 卫星高速过境过程中，地面 UE 如何完成平稳切换；
- 切换过程中的业务连续性如何保持；
- 传统基于信号强度的切换方法在高动态场景下会暴露出哪些问题；
- 后续如何在此基础上引入更合理的联合决策策略。

### 2.2 当前阶段定位

当前阶段的工作重点，已经从“先把基础物理情景构建出来”推进到“将其整理为一个可用于后续毕设实验的 baseline 平台”。

具体来说：

- `2.5` 版本之前，工作重心主要是基础模型构建，包括三维轨道几何、WGS84 地理锚定、Earth-fixed 设计思路等。
- `2.5` 版本之后，工作重点转向平台化与 baseline 收口，即：
  - 从单 UE 原型路径扩展到多 UE baseline；
  - 引入双轨星座与更有代表性的场景；
  - 统一日志、统计、输出和工程结构；
  - 明确当前传统 A3 baseline 的对比口径。

当前研究仓库稳定节点为 `3.1.3`。这里的 `3.1.3` 是研究工作版本，不是 ns-3 框架版本；底层仿真框架仍然是 `ns-3.46`。

## 3. 技术设计一：三维物理模型设计

这一部分是整个仿真研究的物理基础。其核心目的，是把卫星运动、地球自转和地面用户位置统一到一个可信的三维几何框架中，使后续的链路质量、可见性和切换分析具备物理意义。

### 3.1 为什么必须从二维升级到三维

如果仅采用平面近似或二维移动模型，虽然可以快速跑通流程，但难以正确反映以下现象：

- 卫星相对于地球表面的真实过境过程；
- 地面用户视角下的仰角变化；
- 斜距变化与最大仰角不完全同步的情况；
- 多普勒频移的连续演化；
- 不同纬度区域下轨道倾角对过境几何的影响。

因此，本课题前期首先完成的，是从“可跑通的简化模型”向“三维物理可解释模型”的升级。

### 3.2 WGS84 地球模型

地面 UE 的位置不再用简单平面坐标表示，而是采用 WGS84 地球模型中的经纬高来描述，再映射到地心坐标。

这样做有两个意义：

- 第一，地面用户的位置具有真实的地理含义；
- 第二，后续无论是星地距离、仰角、方位角，还是 Earth-fixed 网格划分，都能统一建立在真实地球几何之上。

在课题当前实现中，UE 的默认地面参考点设置在：

- 纬度：`45.6°`
- 经度：`84.9°`
- 高度：`0 m`

这个地理锚点同时也是轨道自动对齐和局部地理网格构建的重要参考点。

### 3.3 ECI 与 ECEF 的角色分工

三维模型中，卫星位置不是直接在地面参考系里“硬移动”，而是经历两层描述：

1. 先在 ECI 坐标系中计算卫星轨道运动；
2. 再转换到 ECEF 坐标系中分析其相对地球表面的运动。

两者的角色如下：

- `ECI`：地心惯性坐标系，用于描述卫星绕地飞行的轨道动力学。
- `ECEF`：地心地固坐标系，用于描述卫星相对于地球表面的可见性、覆盖和服务关系。

简而言之：

- 在 ECI 中，适合算“卫星本身怎么飞”；
- 在 ECEF 中，适合算“卫星相对地面用户在哪里”。

### 3.4 ECI 到 ECEF 的角度转换

ECI 到 ECEF 的转换，本质上是把地球自转纳入卫星位置计算。当前设计中，引入了仿真纪元下的格林威治恒星时，并通过地球绕自身旋转的角度完成从惯性系到地固系的旋转映射。

这一过程的物理意义可以概括为：

- 卫星的轨道是在“宇宙参考系”中演化的；
- 但地面用户处在“随着地球一起旋转的参考系”中；
- 如果不做这一步转换，那么卫星虽然“飞了”，但相对于地面用户的位置关系仍然是不完整的。

因此，ECI 到 ECEF 的转换是后续所有星地相对几何量计算的前提。

### 3.5 三维几何量的计算

在得到卫星的 ECEF 位置和速度之后，课题当前模型可进一步计算以下关键量：

- 卫星到 UE 的斜距；
- 仰角；
- 方位角；
- 多普勒频移。

这些量的作用如下：

- 斜距决定自由空间传播损耗的基本趋势；
- 仰角决定卫星是否进入可见范围，也是邻区与服务判定的重要门限；
- 方位角用于描述卫星在用户天空中的相对方向；
- 多普勒频移反映高动态场景中的频率偏移，是 NTN 场景区别于地面蜂窝的重要物理特征之一。

### 3.6 轨道对齐与过顶控制

为了让仿真场景可控、可解释，前期还设计了轨道对齐逻辑。其目的是让卫星过境过程与目标 UE 所在区域在仿真时间轴上形成可预期的几何关系。

当前实现中引入了以下相关参数：

- `autoAlignToUe`
- `alignmentReferenceTimeSeconds`
- `overpassGapSeconds`
- `interPlaneTimeOffsetSeconds`

这些参数的作用是：

- 控制某颗卫星在指定参考时刻附近与目标区域形成过顶或近过顶关系；
- 控制同轨卫星之间的过境时间间隔；
- 控制不同轨道面之间的相对出现时序。

这一步的意义在于：后续的切换事件不再是随机碰撞出来的，而是建立在可重复、可解释的几何组织之上。

### 3.7 本阶段三维物理模型的总结

本阶段已经完成的核心工作，是将低轨卫星、地球自转和地面用户之间的关系统一到三维空间中，并使仿真具备以下能力：

- 用真实地理锚点描述 UE 位置；
- 用真实轨道演化描述卫星运动；
- 用 ECI/ECEF 转换得到卫星相对于地面的正确位置；
- 用仰角、距离、多普勒等量对链路演化进行解释。

这为后续 Earth-fixed 覆盖设计和切换研究提供了可信的物理底座。

## 4. 技术设计二：Earth-fixed 覆盖与小区设计

在完成三维物理模型之后，下一项关键问题是：卫星的波束究竟如何覆盖地面，以及“波束”和“小区”之间如何建立关系。

这一部分是从“卫星怎么飞”过渡到“网络怎么覆盖”的核心环节。

### 4.1 为什么需要 Earth-fixed 设计

如果只让波束始终追着单个 UE 跑，那么虽然链路看上去很稳定，但会产生两个问题：

- 物理层过于理想化，难以体现真实波束边缘的增益滚降；
- 难以形成地面固定小区，也就难以研究更接近网络规划语义的切换问题。

因此，本课题前期重点梳理了 Earth-fixed 设计思路。其核心不是“让小区跟着卫星移动”，而是：

- 先在地面定义固定的小区区域；
- 再让运动中的卫星通过波束选择与指向控制去服务这些固定区域。

### 4.2 波束与小区的关系

这一部分在汇报中必须明确区分：

- 波束是物理层概念，表示电磁能量在空间中的定向覆盖；
- 小区是网络层概念，表示协议栈上的逻辑服务单元。

在 Earth-fixed 思路下，两者的关系可以概括为：

- 地面先有固定的小区；
- 卫星通过波束去覆盖这些小区；
- 当服务关系变化时，UE 看到的是小区或服务卫星变化，而不是单纯的“光斑扫过”。

### 4.3 基于 WGS84 的局部地理网格

在工程上，不需要对整个地球做全局网格划分。当前设计采用的是：

- 基于 WGS84 的局部研究区域划分；
- 在该区域内生成固定的六边形网格；
- 将这些网格中心作为候选地面锚点。

当前默认网格配置为：

- 网格中心：与 UE 区域对齐；
- 区域范围：`400 km × 400 km`；
- 六边形半径：`20 km`；
- 近邻候选数：`K = 3`。

这意味着，当前 Earth-fixed 设计并不是简单地给每颗卫星绑定一个固定点，而是在研究区域内先建立一个可视化、可搜索、可映射的地面网格目录。

### 4.4 光斑半径与网格尺度设计

为了让网格尺寸具备物理依据，而不是拍脑袋选取，前期对光斑尺寸做了估算。

典型参数为：

- 卫星高度：`600 km`
- 波束半功率宽度：`4°`

在这一参数组合下，地面光斑半径约为 `20 km` 量级。因此，当前仿真中的六边形半径采用 `20 km`，并非随意设定，而是与前期光斑几何分析保持一致。

这样做的意义是：

- 地面网格与物理覆盖尺度基本一致；
- 后续切换边界、邻区竞争和服务关系演化更具解释性。

### 4.5 六边形网格与锚点选择

当前实现中，卫星在运行过程中会根据自身位置选择地面锚点。具体来说：

- 对每颗卫星，根据当前 ECEF 位置寻找其星下点附近的候选网格；
- 从最近的若干个地面网格中选择当前锚点；
- 将该锚点作为卫星当前的地面服务参考位置。

这样，Earth-fixed 设计就不再停留在概念层，而是变成了明确的数据结构和运行逻辑：

- 地面网格是预先生成的；
- 锚点选择是动态更新的；
- 波束和小区的映射关系可以通过日志与导出文件观察。

### 4.6 Earth-fixed 设计的价值

Earth-fixed 设计的价值主要体现在三点：

1. 让覆盖建模具备网络规划语义，而不只是物理追踪；
2. 让后续切换分析更容易映射到固定地面区域；
3. 为后续更复杂的切换策略和资源管理策略提供地理单元基础。

因此，Earth-fixed 不是一个附加细节，而是从“物理可见性建模”走向“网络切换研究”的重要桥梁。

## 6. 当前仿真 baseline 的设计与选择

### 6.1 baseline 的研究定位

当前阶段的 baseline 不是最终优化算法，而是一个对照组。其主要作用是：

- 作为传统切换方法的参考系统；
- 暴露高动态双轨场景下传统 A3 的局限；
- 为后续改进策略提供统一对比口径。

当前 baseline 的定义可以概括为：

> 在固定 `2x4` 双轨、`25 UE`、中等时长仿真场景下，采用仅基于信号质量的 A3 风格切换基线；UE 部署采用“热点增加 + 边界增强 + 背景补充”的二维场景，决策仍只依赖 `RSRP` 比较、`hysteresis`、`TTT` 和邻区可达性约束，不引入负载感知、预测优化或学习型决策。

### 6.2 当前 baseline 的主场景

根据当前稳定节点 `3.1.3`，baseline 主场景为：

- 卫星数：`8`
- 轨道面数：`2`
- 轨道结构：`2x4`
- UE 数：`25`
- UE 部署：`9` 个热点 UE、`10` 个边界 UE、`6` 个背景 UE
- 热点间距：`8 km`
- 边界条带间距：`12 km`
- 跨边界偏移：`5 km`
- 外围背景范围：约 `80 km x 60 km`
- 轨道高度：`600 km`
- 轨道倾角：`53°`
- 轨道面 RAAN 间隔：`6°`
- 轨道面时间偏移：`1 s`
- 参考对齐时刻：`20 s`
- 同轨过境间隔：`4 s`
- 仿真时长：`40 s`

这一配置的目的，不是追求“最大规模”，而是有控制地增强跨轨竞争、多用户空间差异和局部负载聚集，使传统 baseline 更容易暴露问题。

### 6.3 当前 baseline 的切换机制

当前 baseline 采用传统 A3 风格切换语义，主要依据包括：

- `RSRP`
- `hysteresis`
- `TTT`
- 邻区可达性约束

当前默认切换参数为：

- `hoHysteresisDb = 0.2 dB`
- `hoTttMs = 1200 ms`

同时，当前 baseline 引入了严格邻区守卫机制：

- 只有满足可见性和门限条件的邻区，才进入有效候选集合；
- 这样可以避免在多星场景下把明显不合格的候选星纳入切换决策。

### 6.4 当前 baseline 的目的不是“最好”，而是“清楚”

当前 baseline 不是为了证明传统 A3 很优秀，而是为了建立一个清晰、可对比的起点。换句话说：

- baseline 首先要定义清楚；
- 然后才能讨论它有哪些不足；
- 最后才有改进策略的比较价值。

因此，当前阶段的目标不是继续无控制地扩星，而是先把 baseline 的场景、参数、日志口径和统计方式固定下来。

## 7. 平台化与工程化设计

这一部分是中期汇报中非常重要的一环。因为当前工作已经不再是“写一个能跑的 demo”，而是在将研究脚本整理为一个可持续迭代的仿真平台。

### 7.1 主脚本与模块拆分

当前主要仿真入口为：

- `scratch/leo-ntn-handover-baseline.cc`

在工程结构上，已经将部分功能从主脚本中拆分出去，包括：

- `scratch/leo-ntn-handover-runtime.h`
- `scratch/leo-ntn-handover-reporting.h`
- `scratch/leo-ntn-handover-utils.h`

这样做的意义是：

- 主脚本更聚焦于场景搭建与流程控制；
- 运行时状态单独管理；
- 汇总输出和统计逻辑独立管理；
- 通用辅助工具函数统一沉淀。

这表明课题已经从“脚本堆叠”进入“结构化维护”阶段。

### 7.2 运行时状态的工程组织

当前平台已经对卫星侧和 UE 侧的运行时状态做了结构化维护。

卫星侧运行时信息包括：

- 所属轨道面和轨道位置；
- 当前地面锚点；
- 当前邻区集合；
- 已接入 UE 数；
- 汇总业务到达率；
- 负载评分；
- 是否允许接纳新的切入。

UE 侧运行时信息包括：

- 当前初始接入与预测切换状态；
- 切换开始与成功计数；
- 切换执行时延累计；
- 自定义切换逻辑中的当前服务星与候选星状态；
- 历史距离与吞吐统计信息。

这说明平台已不再是一次性脚本，而是具备较完整的运行期上下文管理能力。

### 7.3 日志与统计输出统一

平台当前已统一支持以下输出：

- 切换开始日志；
- 切换成功日志；
- 服务星变化日志；
- 周期性仿真进度输出；
- 最终吞吐统计；
- 最终切换汇总；
- 地理网格目录导出；
- 可选的波束衰减轨迹导出。

其中，结果输出目录统一为：

- `scratch/results/`

这样做带来的好处是：

- 实验结果不再散落；
- 便于保留阶段性 baseline 输出；
- 便于后续做批量对比实验。

### 7.4 多 UE 与双轨场景扩展

从平台角度看，当前一个重要进展是：

- 已从单 UE 路径扩展到多 UE baseline；
- 已支持双轨道面场景；
- 已将关键场景参数暴露为命令行配置项。

这意味着后续研究不必重新推翻已有结构，就可以在统一入口下调整：

- 卫星数；
- UE 数；
- UE 布局类型与热点/边界参数；
- 轨道面数；
- RAAN 间隔；
- 时间偏移；
- 网格参数；
- 切换参数。

这正是平台化设计最核心的价值之一。

### 7.5 稳定性与性能优化

当前平台并不是只关注“功能”，也已经开始处理工程稳定性和性能问题。已做工作包括：

- 纳入周期性仿真进度输出，改善长时间运行的可观测性；
- 做了几何共享计算优化，减少重复轨道传播计算；
- 对切换后可能出现的协议异常加入防御性修正；
- 默认关闭与当前 handover 主线无关的高噪声输出；
- 默认关闭当前 baseline 下易触发无关 PHY 问题的部分 SRS 调度项。

这些工作虽然不直接等于“算法创新”，但对于保证实验可复现、结果可信和后续改进可持续非常关键。

### 7.6 为后续联合策略预留接口

当前 baseline 虽然仍然是纯信号质量驱动，但平台已经预留了负载相关接口，包括：

- `attachedUeCount`
- `offeredPacketRate`
- `loadScore`
- `admissionAllowed`

这意味着平台已经具备了从“纯信号切换”向“信号质量 + 负载”联合决策扩展的工程入口。

因此，当前平台化工作的价值不只是“整理代码”，而是在为后续改进策略搭建统一承载框架。

## 9. 当前平台的意义与不足

### 9.1 当前平台的意义

当前阶段已经形成的，不再只是一个单次验证脚本，而是一个具备以下特点的研究平台：

- 有三维物理底座；
- 有 Earth-fixed 覆盖建模支撑；
- 有多卫星、多轨道面、多用户场景；
- 有明确 baseline；
- 有统一日志、统计和输出结构；
- 有后续策略扩展接口。

这说明课题已经完成了从“前期概念验证”向“中期实验平台搭建”的过渡。

### 9.2 当前仍存在的不足

当前平台虽然已经具备 baseline 研究能力，但仍存在一些需要继续优化的方面：

- baseline 参数尚需进一步冻结与验证；
- 传统 A3 在双轨场景下的问题还需要通过更多实验稳定暴露；
- Earth-fixed 设计与底层协议栈行为之间仍需继续校验一致性；
- 当前负载接口已预留，但尚未正式进入决策过程；
- 场景代表性仍需在可控范围内继续增强。

这些不足并不说明工作停滞，而说明课题已经进入“从平台成型到实验深化”的正常阶段。

## 10. 下一步优化方向

下一阶段的工作建议明确分为三条线同步推进。

### 10.1 baseline 验证线

目标是继续收口并冻结 baseline，使其真正成为后续改进策略的稳定对照组。

重点包括：

- 冻结 baseline 主配置；
- 验证切换事件是否足够稳定；
- 观察是否能持续暴露频繁切换与潜在 ping-pong；
- 统一结果对比口径。

### 10.2 平台增强线

目标是在不破坏对比口径的前提下，继续提升平台的可观测性、稳定性和场景代表性。

重点包括：

- 进一步清理非主线干扰项；
- 保持长时间仿真的稳定性；
- 优化结果导出与批量实验支持；
- 在合理范围内增强双轨和多用户竞争关系。

### 10.3 改进策略准备线

目标是面向毕设任务书，将后续策略从单一信号驱动扩展到联合决策。

重点包括：

- 在当前平台上正式引入负载感知；
- 设计“信号质量 + 卫星负载”的联合指标；
- 在统一 baseline 对照下验证改进策略的收益；
- 重点评估切换成功率、执行时延、吞吐连续性与信令代价。
